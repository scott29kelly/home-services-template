  <action>
    Create the `src/routes/` directory. For each static page, create a route module that:
    1. Imports the existing page component from `../pages/` (re-use, don't rewrite)
    2. Exports a `meta()` function that returns the same metadata the page currently sets via `PageMeta`
    3. Exports the page component as default

    **Pattern for each static route module:**
    ```tsx
    import type { Route } from './+types/{filename}'
    // Import the existing page component
    import ExistingPage from '../pages/ExistingPage'

    export function meta(): Route.MetaDescriptors {
      return [
        { title: 'Page Title | Company Name' },
        { name: 'description', content: '...' },
      ]
    }

    export default function PageRoute() {
      return <ExistingPage />
    }
    ```

    **IMPORTANT:** The existing page components still use `PageMeta` internally. This is fine for now — `PageMeta` uses React 19's native `<title>` hoisting which will produce duplicate titles alongside `meta()` exports. To avoid duplicates, the route module `meta()` export takes precedence. Either:
    - (a) Add the `meta()` export AND keep the existing page's `PageMeta` (slight duplication, harmless), OR
    - (b) Better: Do NOT add `meta()` exports on static routes for now. Let the existing `PageMeta` component continue to work. The `<Meta />` component in root.tsx will render empty if no `meta()` is exported. This avoids any duplicate issue.

    **Choose option (b)** — skip `meta()` exports on static routes. The existing `PageMeta` component handles metadata and works fine during pre-rendering (React 19 renders `<title>` elements server-side). Migrating metadata to `meta()` exports can be done as a follow-up cleanup, NOT a blocker for framework mode.

    So each static route module is simply:
    ```tsx
    import ExistingPage from '../pages/ExistingPage'
    export default function PageRoute() {
      return <ExistingPage />
    }
    ```

    **Feature-flagged routes need special handling:**
