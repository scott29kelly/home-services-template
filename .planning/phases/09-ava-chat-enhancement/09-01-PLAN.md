**1. Create `src/lib/chat-context.ts`**

Export a `PageContext` interface and two functions:

```typescript
export interface PageContext {
  pageName: string
  pageUrl: string
  quickActions: string[]
}
```

**`getPageContext(pathname: string): PageContext`** — Maps pathname to page-specific context:
- Match against `services` array (from `src/config/services.ts`) by comparing slug to `pathname.replace(/^\//, '')` — if a service matches, return `{ pageName: '{service.name} service page', pageUrl: pathname, quickActions: ['Get a {service.name.toLowerCase()} quote', 'Emergency service?', 'How does insurance work?', 'Talk to a real person'] }`
- For known static paths, use a map: `/` -> 'homepage', `/services` -> 'services overview page', `/contact` -> 'contact page', `/about` -> 'about page', `/projects` -> 'projects/portfolio page', `/testimonials` -> 'testimonials page', `/financing` -> 'financing page', `/resources` -> 'resources/blog page', `/ava` -> 'full chat page', `/service-areas` -> 'service areas page'
- For city pages matching `/service-areas/` prefix, return 'service area page' with relevant quick actions
- Default: 'general page' with generic quick actions
- CRITICAL: Every quick action array MUST include `'Talk to a real person'` as the LAST item (per CONTEXT.md locked decision — always visible as a chip)
- Default quick actions for non-service pages: `['I have storm damage', 'How do insurance claims work?', 'Schedule an inspection', 'Talk to a real person']`

**`buildSystemPrompt(pathname: string): string`** — Builds a dynamic system prompt string:
- Import `company` from `src/config/company.ts` and `services` from `src/config/services.ts`
- Call `getPageContext(pathname)` to get the current page name/URL
- Build service list from `services.map(s => s.name).join(', ')`
- Return the system prompt string with this structure (use template literals):
  - Role: "You are Ava, an AI Virtual Assistant for {company.name}."
  - Personality: "Warm, knowledgeable, reassuring, patient. Professional but friendly."
  - Visitor context block: current page name, page URL, available services list, business hours (weekday + saturday), phone number, service area (city, state + "and surrounding areas")
  - Key points: FREE inspections, insurance adjuster meetings, most customers pay only deductible, certifications from `company.certifications.join(', ')`
  - Lead capture instruction: "After a few exchanges when there's a natural opening, offer to have someone reach out. Collect name first, then phone, then email (optional). Keep it conversational — never form-like. If declined, continue helping and try once more later at a natural point. If declined a second time, mention the business phone number or contact page as alternatives."
  - CRITICAL per CONTEXT.md: Do NOT include anything that would make the AI announce page awareness (no "I see you're on the X page"). The context silently guides responses.
  - Response length: "Keep responses concise (2-4 sentences) unless detail is requested."
  - Guardrails: "Never provide specific dollar estimates or guarantee claim approval."

   - No delay for 429 — the rate limit itself is the "wait"

Apply the same pattern to all 3 API files. Changes per file:

**A. `api/chat.js` (Vercel serverless handler)**

1. **Dynamic CORS origin** — Replace `res.setHeader('Access-Control-Allow-Origin', '*')` with:
   ```javascript
   const ALLOWED_ORIGIN = process.env.ALLOWED_ORIGIN || 'http://localhost:5173'
   const origin = req.headers.origin
   const corsOrigin = (origin === ALLOWED_ORIGIN || !process.env.ALLOWED_ORIGIN) ? (origin || ALLOWED_ORIGIN) : ALLOWED_ORIGIN
   res.setHeader('Access-Control-Allow-Origin', corsOrigin)
   res.setHeader('Vary', 'Origin')
   ```
   - When `ALLOWED_ORIGIN` env var is NOT set (local dev), allow any origin (preserves dev experience per research pitfall #4)
   - When `ALLOWED_ORIGIN` IS set (production), only allow that origin
   - Add `Vary: Origin` header (required when Access-Control-Allow-Origin varies by request)
   - Add a comment: `// Set ALLOWED_ORIGIN env var in production to restrict CORS to your domain`

2. **Accept systemPrompt from request body** — Change `const { messages } = req.body` to `const { messages, systemPrompt } = req.body`
   - Use `systemPrompt` if provided, otherwise fall back to the hardcoded `SYSTEM_PROMPT` constant
   - In the Groq API call, change `{ role: 'system', content: SYSTEM_PROMPT }` to `{ role: 'system', content: systemPrompt || SYSTEM_PROMPT }`
   - Add a comment: `// Client sends dynamic system prompt with page context; falls back to default`

3. **Cap incoming messages** — Add `const capped = (messages || []).slice(-10)` and use `capped` instead of `messages` in the API call body. This is a server-side safety cap in addition to the client-side cap.

**B. `api/worker.js` (Cloudflare Worker — Anthropic)**

1. **Dynamic CORS origin** — Replace `'Access-Control-Allow-Origin': '*'` in `corsHeaders` with dynamic origin:
   ```javascript
   const origin = request.headers.get('Origin')
   const allowedOrigin = env.ALLOWED_ORIGIN || 'http://localhost:5173'
   const corsOrigin = (origin === allowedOrigin || !env.ALLOWED_ORIGIN) ? (origin || allowedOrigin) : allowedOrigin
   const corsHeaders = {
     'Access-Control-Allow-Origin': corsOrigin,
     'Vary': 'Origin',
     'Access-Control-Allow-Methods': 'POST, OPTIONS',
     'Access-Control-Allow-Headers': 'Content-Type',
   }
   ```
   - Move `corsHeaders` construction inside the `fetch()` handler (after reading `origin`)
   - Same dev-fallback pattern: when `env.ALLOWED_ORIGIN` is unset, allow any origin

2. **Accept systemPrompt** — Change `const { messages } = await request.json()` to `const { messages, systemPrompt } = await request.json()`
   - Use `systemPrompt || SYSTEM_PROMPT` in the Anthropic API `system` field
   - Cap messages: `const capped = (messages || []).slice(-10)`

**C. `api/worker-groq.js` (Cloudflare Worker — Groq)**

Apply identical changes as worker.js:

3. Cap messages at 10: `const capped = (messages || []).slice(-10)`
4. Use `systemPrompt || SYSTEM_PROMPT` in the `{ role: 'system', content: ... }` message

**All files:** Keep the hardcoded `SYSTEM_PROMPT` constant as a fallback — it ensures the API works even if the client doesn't send a system prompt (backward compatibility). Add a brief JSDoc comment above each `SYSTEM_PROMPT` constant: `/** Default system prompt — overridden by client-sent systemPrompt with page context */`
