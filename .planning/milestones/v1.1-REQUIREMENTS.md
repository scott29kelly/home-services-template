# Requirements — v1.1 Performance & Cleanup

## Goal

Pre-render all routes via React Router 7 framework mode, achieve Lighthouse Performance 90+ on desktop and mobile, and clean up known tech debt from v1.0.

## Why

The template is a client-side SPA serving blank HTML until JavaScript executes. This kills SEO (crawlers see nothing), hurts conversion (slow LCP on mobile), and leaves Lighthouse Performance at 87 desktop / 51 mobile. All content is static and known at build time — there's no reason to render it client-side.

## Approach

Migrate from React Router 7 SPA mode to **framework mode with static pre-rendering**. Every route gets built into real HTML at build time, served from CDN, then hydrates for interactivity. Clean up orphan files and dead code carried from v1.0.

---

## Requirements

### PERF-03: React Router 7 Framework Mode Migration

**What:** Convert the project from SPA mode to React Router 7 framework mode.

**Acceptance criteria:**
- [ ] `react-router.config.ts` configured for framework mode
- [ ] Vite config updated to use React Router Vite plugin
- [ ] Server entry point (`entry.server.tsx`) created
- [ ] Client entry point (`entry.client.tsx`) created or adapted
- [ ] Root route module (`root.tsx`) with document shell (html/head/body)
- [ ] All existing routes converted to route module format
- [ ] Route module exports: `loader`, `meta`, `default` component (as appropriate per route)
- [ ] Data loading moved from client-side hooks/effects to route `loader` functions
- [ ] Config data available via loaders (not direct imports in components where it blocks pre-rendering)
- [ ] `npm run dev` starts the dev server successfully
- [ ] `npm run build` produces a production build without errors
- [ ] All existing functionality preserved — forms, animations, chat, blog, city pages

### PERF-04: Static Pre-rendering (All Routes)

**What:** Configure static pre-rendering so every route is built to HTML at build time.

**Acceptance criteria:**
- [ ] `react-router.config.ts` specifies all routes for pre-rendering
- [ ] Dynamic routes (blog posts, city pages, service pages) enumerated at build time
- [ ] Build output contains `.html` files for every route
- [ ] HTML files contain real, crawlable content (not empty shells)
- [ ] Pages hydrate correctly after initial HTML load (interactive elements work)
- [ ] Vercel deployment config updated for pre-rendered output
- [ ] No client-side loading spinners for initial page content

### PERF-05: Lighthouse Performance 90+

**What:** Achieve Lighthouse Performance score of 90+ on both desktop and mobile.

**Acceptance criteria:**
- [ ] Desktop Lighthouse Performance >= 90
- [ ] Mobile Lighthouse Performance >= 90
- [ ] LCP under 2.5s on desktop, under 4s on mobile
- [ ] CLS under 0.1
- [ ] No render-blocking resources in critical path
- [ ] Above-the-fold content visible in initial HTML (no JS required)

### CLEAN-01: Remove Orphan Files & Unused Exports

**What:** Clean up dead code identified during v1.0 audit.

**Acceptance criteria:**
- [ ] `api/contact.js` removed or properly integrated
- [ ] `api/banner.js` removed or properly integrated
- [ ] `getTestimonialsByService()` removed (confirmed unused)
- [ ] No new unused exports introduced during migration
- [ ] Build succeeds with zero dead-code warnings

### CLEAN-02: Fix or Remove Vitals Handler

**What:** The `vitals.ts` production handler is a no-op placeholder. Either implement real web vitals reporting or remove the dead code.

**Acceptance criteria:**
- [ ] `vitals.ts` either reports to a real endpoint OR is removed entirely
- [ ] If removed: all imports/references cleaned up
- [ ] If kept: reports Core Web Vitals (LCP, FID, CLS) to console or analytics
- [ ] No dead code paths remain

---

## Success Criteria

1. **All routes pre-rendered** — `npm run build` outputs HTML for every route
2. **Lighthouse 90+** — both desktop and mobile Performance scores
3. **Zero tech debt items** — orphan files, unused exports, and vitals placeholder resolved
4. **Feature parity** — everything from v1.0 still works (forms, chat, animations, blog, city pages)
5. **Clean build** — no warnings, no dead code, no orphan files

## Out of Scope

- Server-side rendering (SSR) for dynamic per-request content — all content is static
- Edge functions / streaming SSR — overkill for static content
- CDN configuration changes beyond Vercel defaults
- New features or pages — this is a performance + cleanup milestone only
