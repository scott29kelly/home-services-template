**Font preloads in src/root.tsx:**

Add `<link rel="preload">` tags for the 2-3 most critical woff2 font weights in the `<head>` section, right after the comment "Fonts served from same origin via @font-face in index.css". The critical weights for above-the-fold content are:

1. `Inter 400` (body text — used everywhere)
2. `Plus Jakarta Sans 800` (hero heading — font-extrabold maps to weight 800)
3. `Inter 600` (subheadings and buttons)

The preload links need the actual Vite-hashed asset URLs. Since Vite content-hashes the woff2 files at build time (e.g., `inter-v18-latin-500-Dx4kXJAl.woff2`), we cannot hardcode the hash. Instead, import the font files in root.tsx and use the import URL:

```tsx
// At the top of root.tsx, add font imports:
import interRegularFont from './fonts/inter-v18-latin-regular.woff2?url'
import interSemiboldFont from './fonts/inter-v18-latin-600.woff2?url'
import jakartaExtraboldFont from './fonts/plus-jakarta-sans-v8-latin-800.woff2?url'
```

Then in the `<head>`, after the fonts comment (line 26), add:

```tsx
<link rel="preload" as="font" type="font/woff2" href={interRegularFont} crossOrigin="" />
<link rel="preload" as="font" type="font/woff2" href={interSemiboldFont} crossOrigin="" />
<link rel="preload" as="font" type="font/woff2" href={jakartaExtraboldFont} crossOrigin="" />
```

Note: `crossOrigin=""` is REQUIRED on font preloads per the spec (even for same-origin fonts) — without it, the browser will fetch the font twice (once from preload without CORS, once from @font-face with CORS).

**AVIF hero variant generation in vite.config.ts:**

In the `copyAndOptimizeImages()` plugin, extend the hero image variant generation to also produce AVIF versions alongside the existing WebP variants. Inside the `isHeroImage` block (after the WebP variant generation loop), add AVIF generation:

```typescript
// After the existing WebP heroWidths loop, add AVIF variants:
// Generate AVIF variants (primary format — 30-50% smaller than WebP)
const avifWidths = [768, 1280] // same responsive breakpoints
for (const w of avifWidths) {
  const baseName = path.basename(relPath, ext)
  const avifVariantName = `${baseName}-${w}w.avif`
  const avifVariantPath = path.join(path.dirname(destPath), avifVariantName)
  const avifVariant = await sharp(srcBuffer)
    .resize(w, undefined, { withoutEnlargement: true })
    .avif({ quality: 65 })
    .toBuffer()
  await fs.writeFile(avifVariantPath, avifVariant)
  results.push(
    `  ${path.join(path.dirname(relPath), avifVariantName)}: ${(avifVariant.byteLength / 1024).toFixed(0)}kB (${w}w AVIF variant)`
  )
}
// Full-size AVIF
const fullAvifName = path.basename(relPath, ext) + '.avif'
const fullAvifPath = path.join(path.dirname(destPath), fullAvifName)
const fullAvif = await sharp(srcBuffer).avif({ quality: 65 }).toBuffer()
await fs.writeFile(fullAvifPath, fullAvif)
results.push(
  `  ${path.join(path.dirname(relPath), fullAvifName)}: ${(fullAvif.byteLength / 1024).toFixed(0)}kB (full AVIF)`
)
```

Use `quality: 65` for AVIF — this produces visually equivalent output to WebP quality 80 at significantly smaller file sizes. AVIF's perceptual encoding is more efficient so lower quality numbers yield comparable visual fidelity.

Also add `.avif` to the file extension filter in `collectImageFiles` if not already present (it is — line 47 already includes `avif` in the regex pattern).

**Hero.tsx — Replace img with picture element:**

Replace the current `<img>` tag with a `<picture>` element that serves AVIF as the primary format and WebP as fallback:

1. Add a helper function `heroAvifSrcSet` alongside the existing `heroSrcSet`:

```tsx
function heroAvifSrcSet(src: string) {
  const dot = src.lastIndexOf('.')
  const base = src.substring(0, dot)
  return `${base}-768w.avif 768w, ${base}-1280w.avif 1280w, ${base}.avif 1920w`
}
```

2. Replace the `<img>` element (lines 35-44) with:

```tsx
<picture>

    sizes="100vw"

  />
  <source
    srcSet={heroSrcSet(backgroundImage)}
    sizes="100vw"
    type="image/webp"
  />

    src={backgroundImage}
    alt=""
    role="presentation"
    fetchPriority="high"
    width={1920}
    height={1080}
    className="absolute inset-0 w-full h-full object-cover"
  />
</picture>
```

Note: `fetchPriority="high"` stays on the `<img>` inside `<picture>` — the browser applies it to whichever source is selected.

**root.tsx — Update hero preload to include AVIF:**

Update the existing hero image preload (lines 29-36) to add an AVIF preload alongside the WebP one. The browser will use the format it supports:

Add a new preload link BEFORE the existing WebP preload:

```tsx
{/* Preload LCP hero image — AVIF primary, WebP fallback */}
<link
  rel="preload"
  as="image"
  type="image/avif"
  imageSrcSet="/images/hero-roofing-768w.avif 768w, /images/hero-roofing-1280w.avif 1280w, /images/hero-roofing.avif 1920w"
  imageSizes="100vw"
/>
<link
  rel="preload"
  as="image"
  href="/images/hero-roofing.webp"
  imageSrcSet="/images/hero-roofing-768w.webp 768w, /images/hero-roofing-1280w.webp 1280w, /images/hero-roofing.webp 1920w"
  imageSizes="100vw"
  type="image/webp"
/>
```

Browsers that support AVIF will use the first preload and ignore the second. Browsers that don't support AVIF will skip the first (due to `type="image/avif"`) and use the WebP preload.
